# AI-Spec Language v0.2 — 中間レビュー
日付: 2026-03-15

## 実装状況

TypeScript プロトタイプが動作する状態まで到達。
- パーサー (Lexer → Parser → Resolver → Evaluator → Emitter) の7フェーズパイプライン完成
- CLI (parse / validate コマンド) 実装済み
- テスト 67件 全パス
- サンプル 10件 の .aispec → JSON 変換を確認

## 「効く」ポイント

### 1. `_aispec` メタデータによるトレーサビリティ
出力 JSON に `version`, `parsed_from`, `conditions_evaluated` が自己記述される。
AI が JSON を受け取ったとき「どこから来たか」「どの条件で生成されたか」が即座にわかる。
AI 間のハンドオフ時の混乱防止に有効。

### 2. `@type` + `constraints` の構造化
型情報と制約が分離されており、AI がコード生成・バリデーション生成をする際にパースしやすい。
JSON Schema との親和性もある。

### 3. 条件分岐で環境別 JSON を1ファイルから生成
1つの .aispec から production/staging/development の JSON を出し分けられる。
CI パイプラインで AI が環境ごとのコード生成をする際に便利。

## 「足りない」ポイント

### 1. `_meta` / `ai_hints` に意味論がない
パーサーは `_meta.ai_hints` を単なる JSON キーとして素通しする。
AI が「intent: real-time-notification」を見て何をすべきか、のプロトコルが未定義。
構造はあるが、約束がない状態。

### 2. `@type` の `values` が文字列のまま → 【修正済み】
`values="JPY,USD,EUR"` がカンマ区切り文字列で出力されていた。
→ Emitter を修正し、カンマ区切り文字列を自動的に配列 `["JPY","USD","EUR"]` に変換するようにした。

### 3. 出力 JSON に「何をしてほしいか」がない
現在の JSON は「データスキーマの記述」であり「作業指示書」ではない。
AI 間連携には以下が必要:
- アクション指示（「このスキーマからバリデーターを生成せよ」）
- 依存関係（「先に認証サービスを構築してからこれを実装せよ」）
- 成果物の期待形式（「TypeScript の型定義を返せ」）

### 4. `$` プレフィックスのドメインスキーマに意味がない
`$events`, `$security` などをパーサーは通常の JSON キーと同じに扱う。
`$` を見て特別な処理をする仕組み（スキーマレジストリ）が未実装。

## 総合評価

| 観点                     | 評価 | 理由                                       |
|--------------------------|------|--------------------------------------------|
| 人間→AI の仕様伝達       | ◎    | 構造化されておりパースしやすい               |
| AI→人間 の結果報告       | ○    | `_aispec` メタデータで追跡可能               |
| AI→AI の作業指示         | △    | スキーマはあるがアクション定義がない         |
| AI→AI のパイプライン     | △    | 出力は純粋 JSON で渡せるがプロトコルが未定義 |

## 次のステップ候補

1. `@action` ディレクティブ — 出力 JSON に「この仕様で何をすべきか」を埋め込む
2. `$` プレフィックスのスキーマレジストリ — `$events` を見たら特定構造を期待するルール定義
3. 出力フォーマットのバリエーション — JSON Schema / OpenAPI / TypeScript 型定義への変換
4. AI エージェント間プロトコルの設計 — 仕様受け渡しの標準的なフロー定義
